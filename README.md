# CS-230
CS 230 Game Design 
The client for this project was a company called The Gaming Room‚ who wanted to make a web application/ site out of their existing game Draw It or Lose It․ The existing game was available on Android devices‚ but the company wanted to make a version of it that ran on multiple operating systems․ The main focus was to create a scalable system that could support multiple users on various platforms․

I feel one of my greatest contributions in the development of this documentation is my clear analysis of different operating platforms and the strengths and weaknesses of each one․ I set out with the goal of having a clear analysis of Windows and Linux systems compared to mobile platforms from both a technical and business perspective․ This provided me with the opportunity to make recommendations that were technically sound‚ yet practical for the client․

For me the design document helped while writing code because it made me think ahead before coding․ I had to think about how I would structure the system and how the system components would communicate‚ as well as how the user would use the application‚ instead of just coding it․ This set a structure for the rest of the game to follow‚ which made development smoother and easier to plan․

If I could change one thing, I'd focus on improving the level of detail in my system architecture descriptions․ While I discussed the overall architecture‚ I would have liked to provide more information about how components communicate‚ particularly in terms of scalability and performance․ It would also help if there were some diagrams or detailed examples to help the reader understand

I decided that the system had to be able to handle multiple players at the same time‚ and not impair performance or reliability․ I had to think of my architecture as being cross-platform too‚ because of the client's desire to build apps for platforms beyond Android‚ yet‚ we need to understand the needs of the user before we get too far․ The application is only successful to the extent that it solves the user's problem․ If we design for where we think the problem is‚ and the user has a different problem‚ it won't work․

During the software development‚ I would break the problem up into pieces‚ understand each piece‚ and consider the operating systems and other cloud computing solutions‚ as well as the scaling and performance requirements of the entire system․ From that point‚ I would continue to use this formula in combination with prototyping and user experience feedback to sharpen the design of future projects․

In general‚ I realized that planning and designing are as important to a software project as the coding․ Coding is just part of the process, and a well-designed system is a must if you want to create a good application․
